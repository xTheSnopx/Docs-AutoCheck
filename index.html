Estado del Proyecto AutoCheckAML

Resumen de hitos logrados, arquitectura de seguridad basada en roles, metodologías de prueba implementadas y resolución de incidencias clave.

🚀

¿Qué está funcionando actualmente?

Hemos consolidado la infraestructura base e implementado una lógica de diferenciación visual que asegura un control de acceso granular:

Backend (Servicios Centrales)

  • Base de datos dinámica (SQLite): Inicializada de forma automática con las tablas de control (usuarios, roles, permisos y formularios).
  • Generación de credenciales (Seeds): Carga por defecto de roles específicos e inyección segura de hashes de contraseñas.
  • Autenticación JWT segura: Generación y validación de tokens de acceso para autenticación sin estado y encriptada.

Frontend (Interfaz de Usuario)

  • Modo Operador (Ingeniero Mecánico): Acceso al cuestionario completo de inspección de sistemas del vehículo con firma digital táctil.
  • Filtro HSQ (Seguridad Industrial): Oculta secciones de mecánica avanzada, permitiendo enfocarse en licencias, documentos y seguridad general (extintores, botiquines).
  • Modo Auditoría (Cuadrilla): Carga el formulario en modo lectura inmutable con banners y botón de verificación final.
🛠️

¿Cómo hemos llevado a cabo el desarrollo y pruebas?

Hemos implementado una metodología enfocada en la rapidez de desarrollo y validación cruzada en dispositivos:

  • Servidor de Desarrollo Local: Iniciamos la API del Backend con .NET Core y la UI con React + Vite de forma simultánea, posibilitando pruebas en tiempo real.
  • Validación Móvil mediante IP Local: Configurando los servicios para escuchar en todas las interfaces de red (`0.0.0.0`) y asignando el cliente API a la IP de la máquina local (`192.168.40.5`), permitiendo que el equipo de QA y desarrollo pueda probar la interfaz y su responsividad directamente en un celular real conectado al mismo Wi-Fi.
  • Automatización de Capturas: Utilizando agentes automatizados de navegación (Playwright) para testear los flujos de inicio de sesión y registrar la apariencia visual del sistema en vistas móviles e híbridas.
✅

Errores Encontrados y Soluciones Aplicadas

Durante esta fase de integración, identificamos y solucionamos las siguientes incidencias críticas:

Incidencia Detectada Estado Solución Técnica Aplicada
Claves pre-cargadas en el Login
Las credenciales de Admin aparecían escritas por defecto en el cliente. 		Solucionado Modificación del estado del formulario de Login para inicializar los campos en blanco (`""`), asegurando que cada operador deba escribir obligatoriamente su clave.
Ausencia de Cuentas de Prueba
No había usuarios con roles limitados (HSQ, Cuadrilla) creados por defecto. 		Solucionado Inyección de un cargador lógico en el arranque del servidor (`Program.cs`) que verifica y siembra de forma automática los usuarios `ingeniero`, `hsq` y `cuadrilla` con contraseñas encriptadas.
Desbalance de etiquetas JSX
Errores de sintaxis al inyectar bloques condicionales en el formulario. 		Solucionado Re-estructuración de las etiquetas de cierre en el HTML reactivo de `FormComponent.jsx`, cerrando correctamente los contenedores de botones y grupos interactivos dentro de fragmentos lógicos.
